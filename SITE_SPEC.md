## 2. 設計方針の要約

### 2-1. サイトコンセプト

> **「定式化から、動く仕組みまで。」**
> ― 数理・データ・AIを、ひとつの流れとして扱う会社

**選定理由**

既存HPの情報は一見すると「保険数理」「機械学習」「生成AI」「システム開発」「業務自動化」「教育」と幅広く、そのまま並べると何の会社か伝わりにくくなります。しかし既存HPの3ステップ（問題を定式化する → 説明できるAIを使う → 動くものをつくる）を軸に置くと、これらはすべて**ひとつの課題解決プロセス上の異なる工程**として整理できます。

- 保険数理・確率統計・最適化 = **定式化**の技術
- 機械学習・生成AI・認知科学 = **解く**技術
- システム開発・業務自動化・クラウド = **動かす**技術
- 教育・講師・講演 = 上記の知見を**伝える**活動

つまり「幅広い事業を持つ会社」ではなく、「**課題を数理で定義し、AIで解き、現場で動く仕組みにするまでを一社で担う会社**」として見せます。教育事業も別事業ではなく、「定式化と実装の知見を外部に伝える活動」として同じ軸上に配置します。

### 2-2. 中心メッセージ（優先順位順）

1. **数理とデータで、企業の課題を解決する会社である**（既存HPの中心メッセージを維持）
2. **答えを出す前に、問題を定式化する**（他社との差別化点。既存HPが最も強調している姿勢）
3. **AIは数理の延長。中身を説明できる仕組みを提案する**（ブラックボックス型AIとの違い）
4. **分析で終わらせず、システム・業務自動化として実装する**（実装力）
5. **金融の実務経験（保険数理・データサイエンス組織構築）を持つ**（信頼の根拠）
6. **相談〜開発〜自動化〜教育まで対応できる**（対応範囲）
7. **まだ形になっていない段階の相談も受け付ける**（問い合わせハードルを下げる）

### 2-3. 参考サイトから採り入れる設計要素（表現のみ・内容は非流用）
- 大きな見出し（3行程度）＋短い説明文で始まるページ冒頭
- 事業を3〜4枚のアイコン付きカードで横並びに見せる構成
- 事業同士のつながりを図解で1ブロック割く構成
- 十分な余白と、セクションごとの背景色の切り替え

### 2-4. 参考サイトから採り入れないもの（明確な禁止）
- グループ企業／ホールディングス構造の表現（TSアセットマネジメント合同会社は単体の合同会社）
- 「グループシナジー」という見出し（→「事業のつながり」に置換）
- 上場・従業員規模・成約実績などのスケール表現
- 文章、画像、ロゴ、HTML、CSS、JavaScript の流用

### 2-5. 表現上の禁止事項（法令・誤認防止）
- 金融商品の販売・媒介・投資助言を行っているかのような表現を使わない（「資産運用」は**コンサルティングの対象領域**としてのみ記載）
- 運用成果・利益・効果を保証する表現を使わない
- 「業界No.1」「必ず成果が出る」等の誇大表現を使わない
- 既存HPにない実績・数字・取引先・資格・受賞歴を書かない
- 存在しない部署・子会社・サービス名を作らない
- 「約80%削減」等の数値は、既存HPの記載どおり**齋藤悠貴の個人実績**として、文脈を変えずに記載する（会社の平均実績のように書かない）

## 3. サイトマップ

```
index.html（1ページ）
│
├─ 00 ヘッダー（固定 / グローバルナビ）
├─ 01 ファーストビュー #hero
├─ 02 私たちについて #about
├─ 03 課題解決の進め方 #approach        （問題を定式化する / 説明できるAIを使う / 動くものをつくる）
├─ 04 事業領域 #business                （4領域カード）
├─ 05 事業のつながり #connection        （事業連携図解）
├─ 06 わたしたちの強み #strength        （4項目）
├─ 07 メンバー #members                 （2名 / 経歴アコーディオン）
├─ 08 会社概要 #company                 （会社概要表 ＋ お知らせ ＋ 定款事業内容トグル）
├─ 09 お問い合わせ #contact             （mailto）
└─ 10 フッター
     └─ ページトップへ戻るボタン（固定）
```

### 順序の設計意図
- **私たちについて → 課題解決の進め方 → 事業領域** の順にすることで、「何の会社か」→「どう進めるか」→「何ができるか」と抽象から具体へ降りる。参考サイトのように事業カードを先に出すと、事業数が多い本社では散漫に見えるため、先に**進め方（3ステップ）**を置いて軸を通す。
- **事業のつながり**を事業領域の直後に置き、「4領域はバラバラではない」ことをその場で解消する。
- **強み → メンバー**の順にし、強みの根拠が直後のメンバー経歴で裏づけられる流れにする。
- **お知らせは独立セクションにせず会社概要内**（理由は12章）。
- CTAは FV / 事業のつながり直後 / お問い合わせ の3箇所。

### グローバルナビ項目
`私たちについて` / `課題解決の進め方` / `事業領域` / `メンバー` / `会社概要` / `お問い合わせ（ボタン）`

### 01. ファーストビュー #hero

- **目的**: 3秒で「数理とデータで企業の課題を解決する会社」だと理解させる。交流会や紹介で来訪した人が最短で事業像を掴めるようにする。

  > 企業の課題を解決する。
- **サブコピー案（採用案）**:
  > 保険数理・確率統計を土台に、事象をモデルとして記述します。
  > 答えを出す前に、まず問題を定式化することを大切にしています。

- **補助キーワード**: `保険数理` `確率統計・最適化` `機械学習` `生成AI` `システム開発` `業務自動化（DX）` `教育・講師`

- **背景ビジュアル**: 写真は使用しない。白〜極薄ブルーのグラデーション背景 ＋ CSS/SVGによる**格子（グリッド）レイヤー** ＋ 右側に**正規分布曲線と散布点を模した抽象SVG**（数理・統計の示唆）。ロイヤリティ問題が起きず、GitHub Pagesで軽量。
- **図形・アニメーション**: 読み込み時にメインコピーが2行に分かれて下からフェードアップ（各行 60ms差、計 700ms）。SVGの曲線は `stroke-dasharray` によるドロー（1.2s, ease-out）。散布点は1回だけ小さくフェードイン。ループアニメーションは行わない。

### 9-2. フォントと文字サイズ

#### フォントファミリー

```css
:root{
  --font-ja: "Noto Sans JP", "Hiragino Sans", "Hiragino Kaku Gothic ProN",
             "Yu Gothic", Meiryo, system-ui, sans-serif;
  --font-en: "Inter", "Helvetica Neue", Arial, sans-serif;
  --font-num: "Inter", "Roboto Mono", ui-monospace, monospace;
}
body{ font-family: var(--font-en), var(--font-ja); }
```

- **日本語**: Noto Sans JP（Google Fonts）。ウェイトは 400 / 500 / 700 のみ読み込む。
- **英数字**: Inter（Google Fonts）。ウェイト 400 / 600。英字ラベル（About, Business 等）と数値に使用。
- **候補（代替）**: 日本語 = Zen Kaku Gothic New（やや柔らかい印象）／ M PLUS 1p。英数字 = Manrope（幾何学的で先進性が高い）。**推奨は Noto Sans JP × Inter**（可読性と読み込み速度のバランス）。
- **読み込み方法**: `<link rel="preconnect">` ＋ `display=swap`。サブセット指定で日本語は `&text=` を使わずウェイトを絞ることで軽量化。

## 11. アニメーション設計

企業サイトとして**「気づかれない程度」**を上限とします。派手な動きは信頼性を損ねるため使いません。

| 対象 | 動き | 時間 / イージング |
|---|---|---|
| FV テキスト | 下から 16px フェードアップ。行ごとに 60ms ディレイ | 700ms / `cubic-bezier(.22,.61,.36,1)` |
| FV SVG（分布曲線） | `stroke-dasharray` によるドロー（1回のみ、ループなし） | 1200ms / ease-out |
| FV 散布点 | 順にフェードイン（各30ms差） | 600ms / ease-out |
| スクロール表示（全セクション共通） | `opacity 0 → 1`、`translateY(20px) → 0`。IntersectionObserver（threshold .15、`rootMargin: 0px 0px -10% 0px`）で `.is-visible` 付与。**1回のみ発火**（`unobserve`） | 600ms / ease-out |
| カード群（事業領域・ステップ） | 上記に加え 80ms のスタッガー（CSS変数 `--i` で `transition-delay: calc(var(--i)*80ms)`） | — |
| ヘッダー | スクロール80px超で背景白＋影＋文字色変更（`.is-scrolled`）。scroll イベントは `requestAnimationFrame` でスロットル | 240ms / ease |
| ハンバーガーメニュー | アイコンの3本線が×に変形。オーバーレイは上からスライドイン | 300ms / ease |
| カードホバー | `translateY(-4px)` ＋ `box-shadow: 0 8px 24px rgba(15,42,68,.08)` ＋ 枠線色を `--c-sub` に | 220ms / ease |
| ボタンホバー | 背景を1段濃く（主）／背景に薄色フィル（副）。拡大縮小はしない | 200ms / ease |
| リンクホバー | 下線が左から伸びる（`background-size` 遷移） | 240ms / ease |
| 図解（事業のつながり） | ノードが順にフェードイン、矢印は `stroke-dashoffset` でドロー。スクロールイン時1回のみ | ノード 400ms×4 / 矢印 800ms |
| ページトップボタン | 600px超でフェードイン | 240ms / ease |
| アコーディオン（経歴） | `grid-template-rows: 0fr → 1fr` で高さアニメーション | 300ms / ease |
| ページ内アンカー | `scroll-behavior: smooth` ＋ `scroll-padding-top: 88px`（ヘッダー分） | — |

### スマートフォンで簡略化する動き
- FV の SVG ドローは省略（静的表示）。
- カードのスタッガーを 80ms → 40ms に短縮、または一律表示。
- ホバー系はすべて無効（`@media (hover: hover)` で囲む）。
- 事業のつながり図のアニメーションは行わない（CSS縦積み図はフェードインのみ）。

### prefers-reduced-motion への対応

```css
@media (prefers-reduced-motion: reduce){
  *,
  *::before,
  *::after{
    animation-duration:.01ms !important;
    animation-iteration-count:1 !important;
    transition-duration:.01ms !important;
    scroll-behavior:auto !important;
  }
  .reveal{ opacity:1 !important; transform:none !important; }
}
```
- JS側でも `window.matchMedia('(prefers-reduced-motion: reduce)').matches` を判定し、その場合は IntersectionObserver による付与を行わず、初期状態で `.is-visible` を全要素に適用する。
